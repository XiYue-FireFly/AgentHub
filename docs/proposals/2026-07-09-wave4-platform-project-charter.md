## 1. 立项背景

### 1.1 产品现状

AgentHub 已是可用的**本地多 Agent 协同桌面工作台**（Electron + React + TypeScript），覆盖：

- 本地 CLI Agent 检测与调度  
- API 厂商直连与代理  
- Git 工作台、终端 PTY、SDD 需求闭环  
- MCP / Skills / 备份 / 预算  

`grok` / `groknew` 分支已完成两轮深度 bug 修复与中端功能继实现（旁支线程、GitHub PR 条、备份 UI、敏感路径扩展等）。当前产品处于 **2.0 可发布硬化态**，下一步应从「单机工作台」走向「可扩展平台」。

### 1.2 为什么现在立项

| 驱动 | 说明 |
|------|------|
| 能力天花板 | 插件仅有本地 manifest，缺发现、签名、沙箱 |
| 多机场景 | 配置与备份仍本机为中心，跨设备同步未产品化 |
| 编排可视化 | Firefly/DAG 有运行时，缺团队模板与可视化增强 |
| 自动化入口 | 仅桌面 UI；CI/无人值守缺 headless CLI |
| 合规与可达性 | 无系统级 a11y 审计与无障碍基线 |

若不立项收口范围，容易变成无止境的「顺手加功能」，与 2.0 稳定性目标冲突。

### 1.3 立项目标（一句话）

> 在不破坏 2.0 桌面主路径的前提下，规划并分期交付 **平台扩展层**（插件、同步、编排、无头调度、无障碍），使 AgentHub 可从「个人工作台」演进为「可治理的本地 Agent 平台」。

## 2. 范围定义

### 2.1 本立项包含（In Scope）

| 子项目 | 代号 | 摘要 |
|--------|------|------|
| P1 插件市场与安全沙箱 | `W4-PLUGIN` | 插件发现、清单校验、签名/哈希、安装来源隔离 |
| P2 多机配置同步 | `W4-SYNC` | 加密配置/偏好跨机同步或导入导出协议 |
| P3 Firefly 团队模板与可视化 | `W4-FIREFLY` | 团队编排模板库 + DAG/角色流可视化增强 |
| P4 Headless CLI 调度 | `W4-CLI` | 无 UI 的调度/任务入口（CI / 脚本） |
| P5 无障碍与 i18n 基线 | `W4-A11Y` | WCAG 相关审计 + 键盘/读屏关键路径 |

### 2.2 本立项不包含（Out of Scope）

- 重写 Hub 编排内核或替换 Electron  
- 云端托管多租户 SaaS  
- 主依赖大版本强制升级（Node/Electron 大升单独立项）  
- 无边界的插件任意代码执行（始终 manifest/沙箱优先）  
- 在本分支直接大规模改 `src/**` 实现（本分支以**立项与规格**为主）

### 2.3 与已完成工作的边界

| 已完成 | 本立项关系 |
|--------|------------|
| Wave1 F-N1…F-N7 | 安全/UX 硬化，作为平台基线 |
| Wave2 F-W1…F-W8 | 中端功能继实现，已关闭 |
| 本立项 Wave4 | **新平台能力**，需独立里程碑与验收 |

## 3. 问题陈述与用户价值

### 3.1 用户画像

1. **重度个人开发者**：多机（台式/笔电）、多仓库、多 Agent  
2. **小团队负责人**：希望统一「评审/实现/验证」模板  
3. **平台/DevOps**：希望 CI 中调用 AgentHub 调度能力  

### 3.2 核心痛点

| 痛点 | 影响 | 对应子项目 |
|------|------|------------|
| 插件难发现、不敢装 | 生态难形成 | P1 |
| 换机重配 API/Agent | 上手成本高 | P2 |
| 编排靠口头约定 | 协作不稳定 | P3 |
| 只能点 UI 跑任务 | 无法进 CI | P4 |
| 键盘/读屏支持不完整 | 合规与效率 | P5 |

### 3.3 成功时用户能做到什么

1. 从可信源安装带签名校验的插件，失败有明确原因  
2. 在机器 B 恢复（或同步）加密后的配置与工作区偏好  
3. 一键套用「Firefly 五角色」等团队模板，并在 UI 看清 DAG  
4. 在 CI 中执行：`agenthub run --workspace … --mode orchestrate --prompt …`  
5. 主要工作台路径可用键盘完成，并通过基础 a11y 检查清单

## 5. 技术方案概要（不绑定最终实现）

### 5.1 架构原则

1. **桌面主路径不变**：平台能力以可选模块接入，失败降级到 2.0 行为  
2. **安全默认拒绝**：插件、同步、CLI 均默认最小权限  
3. **契约优先**：IPC / CLI 参数进入 `ipc-contract` 或等价 schema  
4. **可测**：每个子项目至少有单测 + 一条 E2E 或集成脚本  

### 5.2 建议模块边界（未来目录，仅规划）

```text
src/main/platform/
  plugins/     # 市场、校验、安装
  sync/        # 导出导入、同步适配器
  cli/         # headless 入口（或 packages/cli）
src/renderer/platform/
  plugins-ui/
  sync-ui/
  firefly-templates/
```

### 5.3 关键依赖决策（待 ADR）

| 决策点 | 选项 | 倾向 |
|--------|------|------|
| 插件签名 | 自建 ed25519 / Sigstore / 仅哈希 | 先哈希+发布者指纹，签名二期 |
| 同步通道 | 纯文件包 / WebDAV / 自建服务 | 先加密文件包，通道可插拔 |
| CLI 打包 | 独立 npm bin / electron 同源 | 先独立 Node CLI 调本地服务或直连 core |
| Firefly 图 | 自绘 SVG / 引入轻量图库 | 优先自绘或现有 RunTimeline 扩展 |

## 7. 风险与缓解

| 风险 | 级别 | 缓解 |
|------|------|------|
| 插件执行面扩大导致 RCE | 高 | 默认禁止任意 JS；仅 manifest 贡献；沙箱进程可选 |
| 同步包泄露 API Key | 高 | 保持 store 加密形态；导出再加密；禁止明文备份默认开启 |
| CLI 与桌面状态双写冲突 | 中 | 单写入者锁 / 文档约定「桌面运行时 CLI 只读」 |
| Firefly 可视化范围蔓延 | 中 | 只读可视化优先；编辑器二期 |
| 无障碍改造牵动 UI 过大 | 中 | 先关键路径（Composer、会话列表、设置） |
| 人力不足导致半成品 | 高 | 严格按阶段可发布；允许砍 P2 同步仅留导出导入 |

## 9. 验收与度量

### 9.1 产品验收（发布门槛）

- [ ] P1：未校验插件默认不可启用；校验失败可解释  
- [ ] P2：导出包可在第二台机器导入并完成至少 1 次厂商配置恢复  
- [ ] P3：内置 ≥3 模板可应用；DAG 状态与运行一致  
- [ ] P4：CI 示例仓库绿；无显示服务器可 `run`  
- [ ] P5：关键路径键盘闭环；无严重对比度/焦点问题  

### 9.2 工程度量

- `typecheck` / `test` / `build` 全绿  
- 新增 IPC 全进契约校验  
- 安全相关变更有单测或威胁模型条目  

### 9.3 非功能

- 插件安装失败不影响主工作台启动  
- 同步/导入失败可回滚到导入前快照  
- CLI 退出码稳定（0 成功 / 2 校验 / 3 运行失败 / 4 权限）
